# Vercel + PDF 파싱 기술 검증 리포트

> 작성일: 2026-04-20  
> 검토 대상: pdfplumber(Python) vs Node.js 라이브러리를 Vercel에서 운용하는 실현 가능성

---

## 핵심 결론 (한 줄 요약)

**pdfplumber(Python) + Vercel 조합은 기술적으로 가능하나 프로덕션 운용에 적합하지 않음.**  
→ **Node.js 기반 `unpdf` 또는 `pdf-parse` + Vercel Blob 조합 권장**

---

## 1. Vercel Python Serverless 지원 여부

| 항목 | 내용 |
|------|------|
| 지원 여부 | ✅ 가능 (2026년 3월 기준 Beta) |
| 방식 | `api/*.py` 파일을 Vercel Function으로 자동 인식 |
| 지원 프레임워크 | FastAPI, Flask, Django, WSGI/ASGI |
| 상태 | **Beta** — 안정성 보장 없음 |

---

## 2. 크기 제한

| 구분 | 제한값 | 비고 |
|------|--------|------|
| Python 함수 패키지 크기 | **500 MB** (비압축) | pdfplumber 전체 의존성 ~40MB → 여유 있음 |
| Node.js 함수 패키지 크기 | 250 MB (비압축) | |
| **요청/응답 Body 크기** | **4.5 MB** | ⚠️ 이것이 핵심 병목 |

> **실무 문제**: 의료 PDF는 4.5 MB를 초과하는 경우가 많음. 이 경우 `413: FUNCTION_PAYLOAD_TOO_LARGE` 에러 발생.

---

## 3. pdfplumber Vercel 호환성

| 항목 | 평가 |
|------|------|
| 공식 지원 여부 | ❌ 없음 |
| 실제 배포 사례 | ❌ 공개 문서/사례 없음 |
| 의존성 번들링 | ⚠️ Vercel 번들러와 충돌 가능성 |
| 대용량 PDF 처리 | ❌ 메모리 한계 도달 가능 |
| 복잡한 테이블 | ⚠️ 정확도 저하 가능 |

**pdfplumber의 구조적 한계 (서버리스 환경)**
- 로컬 파일시스템 상시 접근을 전제로 설계됨
- 서버리스의 Ephemeral 파일시스템(임시 저장소, 요청 종료 시 삭제)과 충돌
- 모듈 임포트 실패, 타임아웃, 메모리 부족이 무작위로 발생할 수 있음

---

## 4. 실행 시간(Timeout) 제한

| Vercel 플랜 | 기본 타임아웃 | 최대 설정 가능 | 월 비용 |
|-------------|--------------|---------------|---------|
| **Hobby** | **10초** (변경 불가) | 10초 고정 | **무료** |
| **Pro** | 15초 | **300초(5분)** 까지 설정 가능 | **$20/월** |
| Enterprise | 커스텀 | 무제한 | 별도 협의 |

> **80페이지 이상 PDF**의 pdfplumber 처리 시간: 실측 필요하나 10초 초과 가능성 높음.  
> Hobby 플랜에서는 대용량 PDF 처리 실패 위험 있음.

---

## 5. 파일 업로드 크기 제한 상세

| 방식 | 최대 크기 | 비고 |
|------|-----------|------|
| 직접 API Route 업로드 | **4.5 MB** | 기본값, 설정으로 변경 불가 (Vercel 정책) |
| **Vercel Blob 사용** | **제한 없음** | 브라우저 → Vercel Blob 직접 업로드 후 함수에서 URL로 읽음 |

**Vercel Blob 비용**

| 항목 | 무료(Hobby) | Pro 플랜 포함 |
|------|-------------|--------------|
| 스토리지 | 500 MB | 무제한(종량제) |
| 대역폭 | 1 GB/월 | 무제한(종량제) |
| 추가 비용 | 초과 시 요금 발생 | $0.023/GB 스토리지, $0.10/GB 전송 |

---

## 6. 80페이지 이상 PDF 처리 관련 제약

| 문제 | 원인 | 해결책 |
|------|------|--------|
| 타임아웃 | 10~15초 내 처리 불가 | Pro 플랜 + 타임아웃 연장 설정 |
| 메모리 초과 | PDF 전체를 메모리에 적재 | 페이지 단위 스트리밍 처리 |
| 업로드 불가 | 4.5 MB 초과 | Vercel Blob 으로 업로드 분리 |
| 진행률 표시 불가 | 서버리스는 응답이 완료될 때까지 스트림 없음 | Server-Sent Events 또는 폴링 방식 구현 필요 |

---

## 7. Node.js 대안 라이브러리 비교

| 라이브러리 | 서버리스 적합성 | 테이블 추출 | 한글 지원 | 특이사항 |
|-----------|---------------|------------|----------|---------|
| **unpdf** | ✅ 최적 | 보통 | 확인 필요 | 순수 JS, 네이티브 의존성 없음 |
| **pdf-parse v2** | ✅ 양호 | 보통 | 양호 | `serverExternalPackages` 설정 필요 |
| **pdf2table** | ✅ 양호 | ✅ 특화 | 확인 필요 | 테이블 추출 전용 |
| **pdfjs-dist** | ✅ 양호 | 낮음 | 양호 | Mozilla 공식, 범용 |
| pdfplumber(Python) | ⚠️ 불안정 | ✅ 최고 | 양호 | 서버리스 미검증 |

> **딜레마**: 테이블 추출 정확도는 pdfplumber가 가장 높지만, Vercel 서버리스와 궁합이 최악.

## 8. 비용 시나리오 정리

### 시나리오 A: Hobby 플랜 (무료)
- PDF 크기 4.5 MB 이하만 처리
- 처리 시간 10초 이내 PDF만 가능
- 80페이지 이상 PDF → **처리 실패 위험 높음**
- **적합 용도**: 개발/테스트, 소규모 파일

### 시나리오 B: Pro 플랜 ($20/월)
- PDF 크기: Vercel Blob 병행 시 무제한
- 처리 시간: 최대 300초(5분) 설정 가능
- 80페이지 이상 PDF → **처리 가능**
- **적합 용도**: 실제 보험설계사 업무 환경

### 시나리오 C: Python 백엔드 별도 서버 (Railway/Render)
- Railway 무료 tier: 월 $5 크레딧 제공
- Render 무료 tier: 가능하나 콜드스타트 수십 초
- pdfplumber 그대로 사용 가능
- **적합 용도**: 정확도 최우선, Vercel 제약 회피

## 9. 권장 아키텍처 (수정안)

```
[브라우저]
   │
   ├─── 파일 업로드 ──→ [Vercel Blob] ──→ PDF URL 반환
   │
   └─── 처리 요청 ───→ [Next.js API Route (Node.js)]
                              │
                        Vercel Blob에서 PDF URL로 읽기
                              │
                        unpdf / pdf-parse로 파싱
                              │
                        비즈니스 로직 (TS)
                              │
                        [결과 JSON 반환]
                              │
                    [화면 테이블 / 엑셀 다운로드 / PDF 리포트]
```
